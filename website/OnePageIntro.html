This guide provides a quick walk through the goals of fabric3 and shows how to build a simple Hello World application.

Goals and Vision

Building and running applications is getting harder and harder. Users expect more features from even simple applications and at the same time expect robustness and security when using them. Applications are evolving from monolithic islands to networks of distributed units collaborating as part of a coordinated whole. Developers can use these network services to help provide the features their users want but in doing create a price in complexity that someone, often an administrator, needs to pay.

The goal of fabric3 is to make it easier for developers to build these type of applications and for administrators to run and manage them. We do this in two ways:

  • we provide programming models for developers that allows them to build portions of an application in a controlled environment. These models include not only the Java model from the SCA specification but also support for existing models such as Spring, JSF or EJBs. We also intend to support other languages such as Ruby, Groovy or XQuery so that a developer is free to choose the language/framework that works for them.
  • we provide an administrative view of all the application components currently available and how they interact with each other. This allows developers to select other components to provide services while providing control over how those services are connected at runtime.

The problem here is a complex one but we don't think that needs to impact every developer. This guide will walk you through how we do that.

Simple Java Component

Our application here is meant to provide a traditional "Hello World" message but you don't really need a application framework just to print that out. Instead, we'll define a general network service that will construct a suitable message for us. As we are developing in Java, we'll define a Java interface for that service:

Service Contract
public interface HelloWorld {
  /**
   * Returns a personalized greeting.
   * @param name the name of the person
   * @return a personalized greeting
   */
  String sayHello(String name);
}

We used Java to define the interface here because we're Java developers and having a strongly typed interface we can code against makes things easier for us. If we were using a different programming language such as BPEL or Ruby we would perhaps define the interface in WSDL or not at all. The important thing here is that we have a way to define a contract for the service that both the consumer and provider can agree to. Matching a consumer to a provider is handled by the fabric and it will do whatever is necessary to make the contract work. So, for example, a Java consumer could use a Java interface to call a service provider implemented in BPEL using a WSDL interface.

As it is this interface that defines the contract we can write a test case to define the behaviour expected from an implementation. In this example we can write a fairly simple test case using JUnit:

Interface Test
public class HelloWorldTest {
  private HelloWorld service = new HelloImpl();

  public void testGreeting() {
    assertEquals("Hello World", service.sayHello("World");
  }
}

The principal here extends to service interfaces that are much more complex and which also include conversational and callback behaviour. The interface can also be mocked using a testing framework such as EasyMock so that implementations can be developed and tested independently.

Now that we have tests for the behaviour we can write a component that provides the service:

Component Implementation
public class HelloImpl implements HelloWorld {
  private String greeting = "Hello";

  protected setGreeting(String greeting) {
    this.greeting = greeting;
  }

  public String sayHello(String name) {
    return greeting + " " + name;
  }
}

So far we have developed a simple component that provides a basic implementation of a service. It is pure Java code with nothing that ties it to any particular runtime (for example, there are no framework interfaces to implement or base classes to extend). This is basically the POJO development model supported by other frameworks such as Spring, EJB3 or PicoContainer. In fact, components you have already developed for those environments may well run on our fabric without any modification at all (assuming they are not tied to some feature of the framework).

Portable Code

Implementing a component does not require dependencies on anything provided by the framework. This allows you to write the code once and move it between frameworks. It also allows the fabric to do the same.

Now we have written and tested our component we are ready to make it available to other applications through the service network.

Service Network

The previous section described how a developer can write a simple component in Java which provides the implementation of a service contract. Now the code is done, it is time to make that service avialble to others. We do this in two stages:

  1. By contibuting some packaging artifact that contains runnable code
  2. By creating a live, running instance of a component

Contributing Code

The first step to run a component is to make the code that implements it available to the runtimes that are participating in the network. This can be done in a number of ways, depending on the runtime:

  • it could be installed on the machine
  • it could be deployed to an online repository such as Maven's
  • it could be provisioned automatically to the runtime

In a production environment, the code would normally be packaged in some form of distribution archive such as a JAR file. When the archive is contributed, fabric3 examines it to determine what kind of content it provides (such a POJOs, EJBs, Ruby scripts) building up a repository of these contributions. In a development environment, the contribution would just point to where the code was being developed (for example, the classpath from the IDE) so that it can be used directly without needing to repackage all the time.
