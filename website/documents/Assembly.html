Packaged SCA defined configuration files and possibly implementation specific artifacts that can be deployed to a SCA domain. See SCA Assembly Model Spec section 1.2.

Explanation

The SCA Assembly Model specification defines a set of SCA metatdata documents and a packaging format so that code or configurations can be contributed (i.e deployed and activated) to a SCA domain. The SCA assembly model provides a standard way to describe,configure, and package SCA implementation artifacts.

An analogy to JavaEE would be the following: A SCA composite is to a JavaEE web application as a SCA contribution is to a JavaEE application. A SCA composite definition file defines a set of components and their implementations and how they are wired together. A JavaEE web application descriptor file defines a set of servlet implementations and their mappings to URIs. A SCA contribution bundles a set of SCA artifacts together and the contribution definition file declares which artifacts are contributed to the domain and how. On the same token the JavaEE application descriptor declares which contained JavaEE artifacts are to be deployed to the JavaEE container and how.

Example

A SCA Contribution is an example of an assembly. It is packaged as a ZIP file and contains a sca-contribution.xml file that defines which SCA artifacts in the archive are to be contributed to the domain. It also may contain composite, componentType, etc definition files along with implementation specific artifacts such as Java classes, XML configuration files, etc.
