Introduction

Explanation

intents are abstract requirements from the user (application developer), either expressed in the SCDL or using annotations in the Java PM.

intents can be related to a component's operation in a container or about interaction between components or external systems using services and references

they are called implementation and interaction intents respectively

intents can be natively provided by the implementation or binding types. if they do, they are declared using the bindingType and implementationType elements in definitions.xml. For example a jms binding type would declate it would always support transactions. this means if the user requests transactions on a service that use JMS binding we dont need to do anything more. These are called provided intents

They are further categorized into intents that are always provided and intents that may be provided

The way we interpret this in F3 is ..

first one we ignore and assume they are always provided by the type whether the user wants it or not

the may provided one, we explictly pass to the runtime framework wire attachers for bindings and builders for components

It is upto them how they realize it If the intent is not provided by the implementation/binding type

They should map to a policy set.

To give an example

<component requires="f3:managedTransaction"/>

here f3:managedTransaction is an intent requested by te user

in definitions xml we will have <intent name="managedTransaction"/> and there will be a policy that provides this intent

<policySet name"txPolicy" provides="f3:managedTransaction"/>

here provides refer to the intent it provides
